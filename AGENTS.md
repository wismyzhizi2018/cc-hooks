# Personal Codex Rules

适用于 OpenAI Codex 的 AI 协作规范。

## Language
- 默认使用中文回复，除非用户明确要求其他语言。
- 直接、简洁，先给结论，再补必要细节。

## Working Style
- 先看代码和上下文，再下结论，不凭猜测回答。
- 用户要求检查代码是否满足需求时，默认按代码审计方式回答：先说问题，再说结论。
- 没发现明显问题时，明确说明“当前未发现明显问题”。
- 保持现有项目风格，除非用户明确要求，不要扩大改动范围或顺手重构无关内容。

## Git
- 未经用户明确要求，不要主动执行 `git push`。
- 执行 `git push` 后，必须把远程访问地址发给用户。
- 开发前先检查当前分支、工作区状态和本地/远程差异，再决定是否同步代码。
- 如果当前分支是目标需求分支且工作区干净，优先执行 `git pull --rebase` 同步最新代码。
- 如果当前分支有未提交改动，禁止直接 `git pull`，必须先告知当前状态。
- 未经用户明确同意，不要擅自 `stash`、`reset`、处理冲突或覆盖本地改动。
- 默认禁止直接在 `master`、`main`、`test` 分支上修改代码，除非用户明确要求。
- 需求分支命名固定为：`feature#姓名#月份#T工单号_需求名`。
- 热修复分支命名固定为：`hotfix#姓名#月份#T工单号_问题名`。
- 执行 `git commit` 时，使用规范化多行提交信息，禁止单行提交。
- Windows PowerShell 环境下，禁止使用管道方式把中文提交信息传给 `git commit -F -`，避免提交信息乱码。

## Commit Format
提交信息使用以下格式：

```text
<type>(<scope>): <subject>

<详细描述背景、修改了哪些具体文件和核心逻辑，要求条理清晰>

修改内容：
- <类名/文件名> <具体改动1>
- <类名/文件名> <具体改动2>

特性/影响：
- <特殊的业务逻辑说明或影响范围>

Co-Authored-By: Codex <noreply@openai.com>
```

`type` 只能是：`feat`、`fix`、`docs`、`style`、`refactor`、`test`、`chore`。

## Safety
- 不要回退用户未明确要求回退的改动。
- 发现无关改动时默认忽略，只处理当前任务相关内容。
- 如果改动和现有未提交内容冲突，先指出冲突点。
- 默认最小化改动，只修改用户明确指定的文件或当前需求相关代码。
- 需求明确要求“只改某个文件”时，禁止扩大到其他类、公共基类或无关文件。

## Verification
- 改完尽量做最小必要验证。
- 没法验证时，明确说明原因。
- 区分：语法通过、局部验证通过、完整测试通过。
- 提交前自检：需求是否真的实现、是否有明显 bug、是否影响旧逻辑、是否遗漏接口、脚本或文档更新。
- 涉及真实环境动作时，必须先确认，例如发 IM 通知、调外部接口、批量跑脚本、执行影响数据的命令。

## IM Notification Safety
- 涉及 IM 群消息、联调提醒、提测通知、发布通知、回滚通知时，默认先执行 `dry-run` 预览，禁止直接真实发送。
- 真实发送前，必须先把完整消息正文给用户确认，得到明确同意后才能发送。
- 真实发送后，必须明确说明发送对象、发送内容摘要和接口返回结果。
- 未经用户明确要求，不要自行触发任何群消息、Webhook 消息或批量通知。
- 生成 IM 消息时，优先使用项目内已验证通过的 skill 或稳定脚本链路，避免临时拼接发送。
