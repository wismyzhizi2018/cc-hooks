# Personal Claude Rules

适用于所有项目的 AI 协作规范，由 adim 制定。

## Language
- 默认使用中文回复，除非我明确要求其他语言。
- 直接、简洁，先给结论，再补必要细节。

## Working Style
- 先看代码和上下文，再下结论，不凭猜测回答。
- 如果我让你检查代码是否满足需求，默认按代码审计方式回答：先说问题，再说结论。
- 没发现明显问题时，要明确说明"当前未发现明显问题"。
- 保持现有项目风格，除非我明确要求，不要扩大改动范围或顺手重构无关内容。

## Git
- 未经我明确要求，不要主动执行 `git push`。
- 执行 `git push` 后，必须把远程访问地址发给我。
- 开发前先检查当前分支、工作区状态和本地/远程差异，再决定是否同步代码。
- 如果当前分支是目标需求分支且工作区干净，优先执行 `git pull --rebase` 同步最新代码。
- 如果当前分支有未提交改动，禁止直接 `git pull`，必须先告知我当前状态。
- 如果当前不在正确分支，先按规范切换到目标分支，再决定是否同步代码。
- 未经我明确同意，不要擅自 `stash`、`reset`、处理冲突或覆盖本地改动。
- 默认禁止直接在 `master`、`main`、`test` 分支上修改代码，除非我明确要求。
- 开发前如果分支不符合我的规范，先创建正确分支并回报当前分支名。
- 需求分支命名固定为：`feature#姓名#月份#T工单号_需求名` eg:feature#zhangsan#3月份#T00001_需求xx
- 热修复分支命名固定为：`hotfix#姓名#月份#T工单号_问题名` eg:hotfix#zhangsan#3月份#T00001_问题xx
- 分支中的「姓名」优先使用仓库历史分支已有的拼音，无历史参考时通过 `git config user.name` 获取中文名转拼音小写无空格（如「张三」→ `zhangsan`、「李四」→ `lisi`）。禁止使用系统用户名（`whoami`）、邮箱前缀或其他来源。
- 没有单独的 `bugfix` 分支规范，缺陷默认走 `hotfix`。
- 执行 `git commit` 时，必须使用规范化多行提交信息，禁止单行提交。
- 在 Windows PowerShell 环境下，禁止使用管道方式把中文提交信息传给 `git commit -F -`，避免提交信息乱码。
- 中文多行提交信息必须优先使用多个 `-m` 参数，或先写入 UTF-8 编码文件后再使用 `git commit -F <file>`。
- 如果提交信息包含中文，提交后必须检查最近一次 commit message 是否正常显示，确认无乱码后再继续后续操作。
- 如果发现本次 commit message 出现乱码，必须立即撤销该提交，保留代码改动，查明原因后再重新提交。
- 注意 git commit message 具体对齐参考仓库已有历史风格。

## Commit Format
提交信息必须严格使用以下格式：

```
<type>(<scope>): <subject>

<详细描述背景、修改了哪些具体文件和核心逻辑，要求条理清晰>

修改内容：
- <类名/文件名> <具体改动1>
- <类名/文件名> <具体改动2>

特性/影响：
- <特殊的业务逻辑说明或影响范围>

Co-Authored-By: Claude <noreply@anthropic.com>
```

### Rules
- `type` 只能是：`feat`、`fix`、`docs`、`style`、`refactor`、`test`、`chore`
- `scope` 写工单号、模块名或明确范围
- `subject` 不超过 50 个字
- 提交信息必须结合当前实际改动生成，不能写空泛模板
- 提交信息末尾必须追加 `Co-Authored-By`
- 默认格式为：`Co-Authored-By: Claude <noreply@anthropic.com>`
- 如果当前会话可以明确识别模型名称，则写入真实模型名，例如：`Co-Authored-By: Claude claude-sonnet-4.6 <noreply@anthropic.com>`
- 如果无法确认模型名称，则回退为：`Co-Authored-By: Claude Code <noreply@anthropic.com>`

## Safety
- 不要回退我未明确要求回退的改动。
- 发现无关改动时默认忽略，只处理当前任务相关内容。
- 如果改动和现有未提交内容冲突，先指出冲突点。
- 默认最小化改动，只修改我明确指定的文件或当前需求相关代码。
- 如果发现顺手优化点或可重构点，先记录，不要直接改。
- 需求明确要求"只改某个文件"时，禁止扩大到其他类、公共基类或无关文件。

## Verification
- 改完尽量做最小必要验证。
- 没法验证时，明确说明原因。
- 要区分：语法通过、局部验证通过、完整测试通过。
- 提交前必须自检：需求是否真的实现、是否有明显 bug、是否影响旧逻辑、是否遗漏接口、脚本或文档更新。
- 涉及真实环境动作时，必须先确认，例如：发 IM 通知、调外部接口、批量跑脚本、执行影响数据的命令。

## Coordination
- 涉及前端联调时，默认先检查接口字段是否存在破坏性变更。
- 新增字段、返回结构调整、判空行为变化时，必须明确告知前端。
- 如果需求包含前端接口或文档变更，默认同步检查接口文档是否需要更新。
- 提测时默认整理可复现信息：测试单号、执行命令、预期结果、实际结果。
- 如果只是局部验证通过，必须明确说明未覆盖范围。
- 测试反馈问题后，优先给出根因、修复点、验证方式。
- 涉及运维发布时，默认整理上线内容、影响范围、上线后观察项。
- 如果存在风险点或回滚条件，必须提前说明。

## IM Notification Safety
- 涉及 IM 群消息、联调提醒、提测通知、发布通知、回滚通知时，默认先执行 `dry-run` 预览，禁止直接真实发送。
- 真实发送前，必须先把完整消息正文给用户确认，得到明确同意后才能发送。
- 真实发送后，必须明确说明发送对象、发送内容摘要和接口返回结果。
- 未经我明确要求，不要自行触发任何群消息、Webhook 消息或批量通知。
- 上线、联调、提测相关通知默认沿用项目既有模板，不要擅自改写风格。

## IM Notification Encoding
- 生成 IM 消息时，优先使用项目内已验证通过的 skill 或稳定脚本链路，避免临时拼接发送。
- 禁止使用临时 PowerShell 脚本拼接中文消息正文后直接发送，避免出现 UTF-8/GBK 编码乱码。
- 如果消息包含中文标点、链接、@人信息，发送前必须先做 `dry-run` 检查，确认正文无乱码、无错链、无重复 `@`。

## IM Notification Content
- `text.content` 只负责正文展示，真实 `@` 以手机号参数为准，两者都要检查。
- 通知文案优先对齐项目既有模板和 `WORKING_AGREEMENT` 约定，不要擅自改成泛化模板。

## PowerShell 命令转义
- 在 PowerShell 中执行 `php -r`、`php artisan tinker --execute` 这类内联 PHP 时，优先用单引号包裹整段 PHP 代码，内部字符串再用双引号。
- 如果整条命令必须放在 PowerShell 双引号字符串中，PHP 里的变量如 `$row`、`$id`、`$code` 必须写成 `` `$row ``、`` `$id ``、`` `$code ``，避免被 PowerShell 先做变量插值。
- 不要在 PowerShell 单引号字符串里额外写 `\"` 这类 C 风格转义；单引号里的双引号本来就不需要转义，额外的反斜杠会原样传给 PHP，最终变成语法错误。
- 命令层级复杂时，优先用 here-string 或 UTF-8 临时文件执行，减少多层引号和转义相互污染。
